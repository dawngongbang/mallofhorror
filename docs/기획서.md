# 몰오브호러 온라인 (Mall of Horror Online)
## 프로젝트 기획서 v1.0

> 작성일: 2026-03-23
> 목적: 몰오브호러 보드게임을 서버 없이 친구들과 온라인으로 플레이하기 위한 웹 앱 구현

---

## 1. 게임 소개

### 1.1 원작 개요
몰오브호러(Mall of Horror)는 2006년 발매된 좀비 테마 협상/배신 보드게임이다.
쇼핑몰을 배경으로 2~6명의 플레이어가 각자 3명의 생존자를 이끌며, 매 라운드 쏟아지는
좀비 무리를 피해 살아남는 것이 목표다.

### 1.2 핵심 재미 요소
- **협상과 배신**: 동맹을 맺고, 필요하면 배신한다
- **다수결 투표**: 좀비 구역의 소수파가 희생된다 — 내 생존자를 살리려면 남을 설득해야 한다
- **자원 관리**: 이동 카드, 아이템, 캐릭터 능력 활용
- **긴장감**: 매 라운드 좀비가 늘어나며 쇼핑몰이 점점 위험해진다

## 2. 게임 규칙 요약

### 2.1 기본 구성
| 항목 | 내용 |
|------|------|
| 인원 | 2~6명 |
| 소요 시간 | 60~90분 |
| 목표 | 마지막까지 생존자를 1명 이상 남긴 플레이어 |

### 2.2 쇼핑몰 구역

```
┌──────────────────────────────────┐
│  [주차장]   [슈퍼마켓]  [극장]   │  ← 외곽 (좀비 진입 구역)
│      ↓           ↓         ↓     │
│         [중앙 복도]              │  ← 중간
│      ↓           ↓         ↓     │
│  [카페]    [경비실]    [옥상]    │  ← 내부 (비교적 안전)
└──────────────────────────────────┘
         ↑ 좀비는 외곽부터 침입
```

각 구역은 **최대 수용 인원**이 있으며, 좀비가 침입하면 그 구역의 생존자 중
**소수파**가 좀비에게 잡아먹힌다.

### 2.3 라운드 진행 순서

```
1. 좀비 카드 공개
   └─ 이번 라운드 좀비가 침입할 구역과 수량 공개

2. 협상 페이즈 (타이머: 2~3분)
   └─ 플레이어들이 자유롭게 협상, 거래, 약속

3. 이동 페이즈
   └─ 모든 플레이어가 동시에 이동 카드를 선택
   └─ 카드 공개 (이동은 취소 불가)

4. 투표 페이즈
   └─ 좀비가 있는 구역마다 투표 실시
   └─ 소수 생존자 → 제거 (동수면 추가 규칙 적용)

5. 좀비 이동
   └─ 잔류 좀비가 내부로 한 칸 이동

6. 이벤트 카드
   └─ 특수 효과 적용 (아이템, 캐릭터 능력 포함)
```

### 2.4 승리 조건
- 생존자를 1명 이상 남긴 채 게임이 끝나면 승리
- **게임 종료 조건**: 좀비 덱이 소진되거나, 생존자가 1명만 남으면 종료

## 3. 기술 스택 및 아키텍처

### 3.1 설계 원칙
> "서버가 없어도 된다" — BaaS(Backend-as-a-Service)를 최대한 활용해
> 인프라 관리 없이 실시간 멀티플레이어를 구현한다.

### 3.2 기술 스택

#### Frontend
| 항목 | 기술 | 이유 |
|------|------|------|
| 프레임워크 | React 18 + TypeScript | 생태계, 타입 안전성 |
| 빌드 도구 | Vite | 빠른 개발 서버, HMR |
| 상태 관리 | Zustand | 경량, 게임 상태에 적합 |
| 스타일 | Tailwind CSS | 빠른 UI 구성 |
| 보드 렌더링 | React + SVG / Canvas | 커스텀 게임 보드 |
| 애니메이션 | Framer Motion | 좀비 이동, 투표 결과 |

#### Backend-as-a-Service (서버 불필요)
| 항목 | 기술 | 무료 플랜 한도 |
|------|------|--------------|
| **실시간 DB** | Firebase Realtime Database | 1GB 저장, 10GB/월 전송 |
| **인증** | Firebase Authentication | 무제한 (익명 로그인) |
| **호스팅** | Vercel | 무제한 (취미 프로젝트) |

> **왜 Firebase인가?**
> - 별도 서버 코드 불필요
> - 실시간 동기화가 내장 기능
> - 무료 플랜으로 소규모 플레이 완전 커버
> - 오프라인 지원 내장

#### 대안 스택 (Firebase 대신)
| 상황 | 대안 | 특징 |
|------|------|------|
| SQL 선호 | Supabase | PostgreSQL 기반, 무료 500MB |
| P2P 원할 때 | PeerJS (WebRTC) | 서버 완전 불필요, 연결 불안정 가능성 |
| 심플하게 | Partykit | 게임에 특화된 실시간 플랫폼 |

### 3.3 시스템 아키텍처

```
┌─────────────────────────────────────────────────────────┐
│                    플레이어 브라우저                       │
│  ┌──────────────┐    ┌──────────────┐    ┌────────────┐ │
│  │  게임 UI     │    │  게임 엔진   │    │  Firebase  │ │
│  │  (React)     │◄──►│  (순수 TS)  │◄──►│  SDK       │ │
│  └──────────────┘    └──────────────┘    └─────┬──────┘ │

                                    │   Firebase Realtime DB      │
```

### 3.4 데이터 구조 (Firebase)

```typescript
// /games/{roomId}
interface GameRoom {
  id: string;
  status: 'waiting' | 'playing' | 'finished';
  hostId: string;
  createdAt: number;
  settings: GameSettings;
  players: Record<string, Player>;
  state: GameState;
  chat: Record<string, ChatMessage>;

interface Player {
  id: string;
  nickname: string;
  color: PlayerColor;
  survivors: Survivor[];

  round: number;
  currentZombieCard: ZombieCard;

  votes: Record<ZoneName, Record<string, string>>; // 투표
  timer: number;
}
```

## 4. 주요 기능 명세

### 4.1 방 생성 및 입장

#### 방 만들기
- 닉네임 입력 → 6자리 방 코드 자동 생성
- 게임 설정: 인원 제한(2~6), 협상 시간, 이벤트 카드 ON/OFF
- 방 링크 공유 버튼 (클립보드 복사)

#### 방 입장
- 방 코드 6자리 입력 또는 초대 링크 클릭
- 닉네임 + 색상 선택
- 대기실에서 준비 완료 표시

### 4.2 게임 진행 UI

#### 메인 보드
```
┌────────────────────────────────────────────────┐
│  라운드 3 | 협상 중... ⏱️ 01:45                │
├────────────────────────────────────────────────┤
│                                                │
│   [주차장🧟×3]    [슈퍼마켓🧟×1]  [극장🧟×0]  │
│    👤김철수        👤이영희                    │
│    👤박민수                                    │
│                                                │
│        [중앙복도]                              │
│         👤홍길동   👤최수진                    │
│                                                │
│   [카페]         [경비실]        [옥상]         │
│                   👤나준영                     │
│                                                │
├────────────────────────────────────────────────┤
│ 💬 채팅 / 협상                    [이동 선택]  │
└────────────────────────────────────────────────┘
```

#### 페이즈별 UI
| 페이즈 | 주요 UI |
|--------|---------|
| 협상 | 채팅창 활성화, 카운트다운 타이머, 이동 카드 미리보기 |
| 이동 | 각자 이동 카드 선택 (봉인), "준비 완료" 표시 |
| 투표 | 구역별 투표 카운트 실시간 표시, 결과 애니메이션 |
| 좀비 | 좀비 이동 애니메이션, 희생자 제거 연출 |
| 이벤트 | 이벤트 카드 공개 및 효과 적용 |

### 4.3 협상 시스템
- **텍스트 채팅**: 전체/귓속말 구분
- **공개 약속**: "A가 B에게 이번 라운드 투표 지원" 형태로 공개 선언 (구속력 없음)
- **이모지 반응**: 빠른 감정 표현

### 4.4 투표 시스템
- 좀비가 있는 구역마다 독립 투표
- 각 플레이어는 자기 구역의 생존자 수만큼 투표권 보유
- 실시간으로 집계 현황 표시 (누가 투표했는지는 결과 공개 전 숨김)
- 동수 처리: 추가 규칙 선택 가능 (전원 제거 / 랜덤 / 재투표)

### 4.5 관전 모드
- 탈락한 플레이어 관전 가능
- 전체 게임 상태 열람 (이동 카드 포함)

## 5. UI/UX 설계 방향

### 5.1 디자인 컨셉
- **분위기**: 어두운 톤, 좀비 호러 + B급 감성
- **팔레트**: 딥 레드, 다크 그레이, 낡은 노랑 (경고 색)
- **폰트**: 공포 분위기의 헤드라인 폰트 + 가독성 좋은 본문 폰트

### 5.2 반응형
- **주 타겟**: 데스크탑 (1024px 이상) — 보드게임 특성상
- **모바일**: 기본 플레이 가능 수준 (투표/채팅 위주)

### 5.3 접근성
- 색맹 대비: 플레이어 구분을 색 + 아이콘 조합으로
- 한국어 기본, 영어 지원 고려

### 5.4 핵심 UX 원칙
1. **페이즈가 명확해야 한다** — 지금 뭘 해야 하는지 항상 명시
2. **실수를 방지한다** — 이동 확정 전 확인 다이얼로그
3. **상태를 투명하게** — 누가 준비됐는지, 어떤 페이즈인지 항상 표시
4. **연결 문제를 우아하게** — 재연결 시 상태 자동 복원

## 7. 리스크 및 대응

| 리스크 | 가능성 | 대응 방안 |
|--------|--------|-----------|
| Firebase 무료 한도 초과 | 낮음 | 게임 종료 후 데이터 정리 자동화 |
| 플레이어 연결 끊김 | 중간 | 재연결 시 상태 복원, 60초 후 자동 퇴장 |

| 치팅 (클라이언트 조작) | 낮음 | Firebase Security Rules로 유효하지 않은 상태 변경 차단 |
| 협상 시간 초과 | 낮음 | 타이머 종료 시 자동 다음 페이즈 진행 |
