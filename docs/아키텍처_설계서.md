## 1. 턴제 게임을 Firebase로 구현할 때의 핵심 문제

Firebase Realtime Database는 서버 코드가 없다. 모든 로직이 클라이언트에서 실행된다.
이것이 턴제 멀티플레이어에서 만드는 세 가지 핵심 문제:

| 문제 | 설명 | 해결 전략 |
|------|------|-----------|
| **누가 페이즈를 전환하나?** | 서버가 없으니 타이머를 믿을 수 없다 | 호스트 클라이언트가 권위를 가짐 |
| **동시 행동의 봉인** | 이동 카드 선택을 남들이 보면 안 됨 | Security Rules로 pendingMoves 접근 제한 |
| **Race Condition** | 두 클라이언트가 동시에 같은 필드를 쓸 때 | Firebase Transaction으로 원자적 업데이트 |

### 2.2 페이즈 전환 규칙

```typescript
type GamePhase =
  | 'waiting'
  | 'reveal_zombie'
  | 'negotiation'
  | 'movement'
  | 'reveal_movement'
  | 'voting'
  | 'resolve_votes'
  | 'zombie_advance'
  | 'event_card'
  | 'check_win'
  | 'finished';

// 페이즈 전환 조건 — 순수 함수, 테스트 가능
function canAdvancePhase(state: GameState, players: Record<string, Player>): boolean {
  switch (state.phase) {
    case 'waiting':
      return Object.values(players).every(p => p.isReady) && Object.keys(players).length >= 2;
    case 'reveal_zombie':
      return Date.now() >= state.phaseDeadline;
    case 'negotiation':
      return Date.now() >= state.phaseDeadline;
    case 'movement':
      // 모든 생존자가 있는 플레이어가 이동을 제출했거나, 타이머 만료
      const activePlayers = Object.values(players).filter(p => hasLivingSurvivors(p, state));
      const allSubmitted = activePlayers.every(p => state.moveStatus[p.id] === true);
      return allSubmitted || Date.now() >= state.phaseDeadline;
    case 'voting':
      return allVotesSubmitted(state, players);
    default:
      return Date.now() >= state.phaseDeadline;
  }
}
```

### 2.3 호스트의 역할

호스트 클라이언트는 다음을 전담한다:
1. 타이머 만료 감지 → 페이즈 전환 Firebase 업데이트
2. 이동 카드 공개 (pendingMoves → resolvedMoves)
3. 투표 결과 집계 및 생존자 제거
4. 좀비 전진 처리
5. 이벤트 카드 적용

**호스트가 끊기면?** → 호스트 마이그레이션 (섹션 6.3)

type ZoneName =

### 3.2 게임 상태 타입

```typescript
// === 구역 상태 ===
interface ZoneState {
  zombies: number;
  survivors: Record<string, SurvivorSnapshot>; // survivorId → snapshot
}

interface SurvivorSnapshot {
  survivorId: string;
  playerId: string;
  characterId: CharacterId;
  name: string;
  isAlive: boolean;
}

// === 이동 ===
// ★ 이동 페이즈는 4단계로 구성:
//    1) character_select: 보안관부터 순서대로 이동할 캐릭터를 공개 선언
//    2) destination_seal: 모든 플레이어가 목적지를 동시에 봉인
//    3) destination_reveal: 목적지 동시 공개
//    4) move_execute: 보안관부터 순서대로 이동 처리

// 캐릭터 선언 (공개) — character_select 단계
interface CharacterDeclaration {
  playerId: string;
  survivorId: string;   // 이동할 생존자 (모두에게 공개)
  declaredAt: number;
  order: number;        // 선언 순서 (보안관=0, 다음=1, ...)
}

// 목적지 봉인 (비공개) — destination_seal 단계
interface SealedDestination {
  playerId: string;
  targetZone: ZoneName; // 본인 + 호스트만 읽기 가능
  submittedAt: number;
}

// 공개 후 확정된 이동 — destination_reveal 이후
interface ResolvedMove {
  playerId: string;
  survivorId: string;
  targetZone: ZoneName;
  order: number;        // 보안관 기준 실행 순서
  executed: boolean;
  blockedReason?: 'zone_full'; // 정원 초과로 이동 불가
}

// 보안관 (선플레이어) — 매 라운드 한 칸씩 이동
type SheriffPlayerId = string;

// === 투표 ===
// votes[zoneName][voterId] = targetPlayerId (누구를 희생시킬지)
type VoteMap = Record<ZoneName, Record<string, string>>;

interface VoteResult {
  zone: ZoneName;
  tally: Record<string, number>;    // playerId → 받은 표 수
  loser: string | null;              // 가장 적게 받은 플레이어 (소수파)
  sacrificedSurvivorIds: string[];   // 실제로 제거된 생존자
  tieBreak: 'none' | 'all_eliminated' | 'random' | 'revote';
}

// === 핵심 게임 상태 ===
interface GameState {
  // 페이즈
  phase: GamePhase;
  round: number;
  phaseDeadline: number;      // 이 타임스탬프가 되면 호스트가 다음 페이즈로
  phaseStartedAt: number;     // 디버깅/로그용

  // 덱 (서버 권위 버전 — 순서 중요)
  zombieDeck: ZombieCard[];
  zombieDeckIndex: number;    // 현재까지 뽑은 카드 수
  currentZombieCard: ZombieCard | null;

  eventDeck: EventCard[];
  eventDeckIndex: number;
  currentEventCard: EventCard | null;

  // 보드
  zones: Record<ZoneName, ZoneState>;

  // 라운드별 데이터 (라운드 시작 시 초기화)
  pendingMoves: Record<string, PendingMove>;   // playerId → 봉인된 이동
  resolvedMoves: Record<string, PendingMove>;  // 공개된 이동
  moveStatus: Record<string, boolean>;         // playerId → 제출 여부 (모두에게 보임)

  votes: VoteMap;
  voteStatus: Record<ZoneName, Record<string, boolean>>; // zone → playerId → voted
  voteResults: Record<ZoneName, VoteResult>;

  // 활성 효과
  activeEffects: ActiveEffect[];

  // 결과
  winners: string[];    // 게임 끝났을 때 생존자 있는 플레이어
  eliminatedAt: Record<string, number>; // playerId → 탈락한 라운드
}

type ActiveEffect = {
  cardId: string;
  effect: EventCardEffect;
  expiresAtRound: number;
};
```

### 4.2 Security Rules

```json
{
  "rules": {
    "games": {
      "$roomId": {
        "meta": {
          ".read": true,
          ".write": "auth != null && (
            !data.exists() ||
            auth.uid === data.child('hostId').val()
          )"
        },

        "players": {
          ".read": true,
          "$playerId": {
            ".write": "auth.uid === $playerId"
          }
        },

        "game": {
          ".read": true,

          "pendingMoves": {
            "$playerId": {
              ".read": "auth.uid === $playerId || auth.uid === root.child('games').child($roomId).child('meta/hostId').val()",
              ".write": "auth.uid === $playerId && root.child('games').child($roomId).child('game/phase').val() === 'movement'"
            }
          },

          "moveStatus": {
            "$playerId": {
              ".read": true,
              ".write": "auth.uid === $playerId"
            }
          },

          "votes": {
            "$zoneId": {
              "$playerId": {
                ".read": true,
                ".write": "auth.uid === $playerId && root.child('games').child($roomId).child('game/phase').val() === 'voting'"
              }
            }
          },

          "voteStatus": {
            "$zoneId": {
              "$playerId": {
                ".read": true,
                ".write": "auth.uid === $playerId"
              }
            }
          },

          "$other": {
            ".read": true,
            ".write": "auth.uid === root.child('games').child($roomId).child('meta/hostId').val()"
          }
        },

        # ★ 비공개 데이터 (아이템, 이동 카드 패)
        "private": {
          "$playerId": {
            # 본인과 호스트만 읽기 가능
            ".read": "auth.uid === $playerId || auth.uid === root.child('games').child($roomId).child('meta/hostId').val()",
            # 본인만 쓰기 가능 (호스트는 게임 시작 시 초기 아이템 배분을 위해 쓰기 권한 별도 부여)
            ".write": "auth.uid === $playerId || auth.uid === root.child('games').child($roomId).child('meta/hostId').val()"
          }
        },

        "chat": {
          ".read": true,
          "$messageId": {
            ".write": "auth != null && (!data.exists() || auth.uid === data.child('playerId').val())"
          }
        }
      }
    }
  }

# /private/{playerId}/ 아래 모든 데이터를 본인만 읽게 하면,
# 아이템, 이동 카드 패 등 모든 비공개 정보를 한 곳에서 관리할 수 있다.
#
# 구조:
#   /games/{roomId}/private/{playerId}/
#     items/
#       {itemId}/
#         id: "item_knife"
#         name: "칼"
#         effect: { type: "extra_vote" }
#         acquiredAt: 1742000100000
#     hand/                          ← 이동 카드 패 (아직 내지 않은 카드들)
#       {cardId}/
#         id: "move_card_3"
#         targetZone: "corridor"     ← 카드에 인쇄된 이동 목적지
#     usedItems/                     ← 이번 라운드 사용한 아이템 (공개 가능)
#       {itemId}: true
```

**핵심 포인트**:
- `pendingMoves`는 본인과 호스트만 읽을 수 있음 → 이동 봉인 보장
- `game`의 일반 필드들은 호스트만 쓸 수 있음 → 페이즈 전환 권한 집중
- `votes`, `moveStatus`는 개별 플레이어가 자기 것만 씀

### 5.3 동시 이동 처리 흐름 (상세)

```
MOVEMENT 페이즈 시작
    │
    ├── 호스트: phaseDeadline = now + movementTime 설정
    │
    ├── 각 플레이어 (병렬):
    │   ├── UI: 이동 카드 선택
    │   ├── 선택 완료 → /game/pendingMoves/{uid} 에 씀 (자신만 읽기 가능)
    │   └── /game/moveStatus/{uid} = true 씀 (모두에게 보임)
    │
    ├── 모든 클라이언트: moveStatus 변화 감지
    │   └── 전원 true → 호스트에게만 보임: "이제 공개 가능"
    │
    └── 호스트: canAdvancePhase() === true 감지
        ├── Firebase Transaction:
        │   current_phase === 'movement' 이면:
        │     resolvedMoves = pendingMoves (복사)
        │     pendingMoves = {} (초기화)
        │     phase = 'reveal_movement'
        │     phaseDeadline = now + 3000
        └── (Transaction 실패 시 → 이미 다른 누군가가 전환함, 무시)
```

**왜 Transaction인가**: 호스트가 바뀌는 마이그레이션 순간에 두 클라이언트가 동시에 전환을 시도할 수 있다. Transaction은 원자적으로 처리되어 중복 전환을 방지한다.

### 6.3 호스트 마이그레이션

호스트 클라이언트가 끊기면 게임이 멈춘다. 이를 방지:

```typescript
// 1. 각 클라이언트가 30초마다 lastSeen 갱신
setInterval(() => {
  db.ref(`games/${roomId}/players/${myUid}/lastSeen`).set(Date.now());
}, 30_000);

// 2. onDisconnect 설정 (브라우저가 닫히면 자동 실행)
db.ref(`games/${roomId}/players/${myUid}/isConnected`).onDisconnect().set(false);

// 3. 모든 클라이언트가 호스트 상태를 감시
// 호스트의 lastSeen이 60초 이상 지났으면 → 마이그레이션 시작
subscribeToPlayers(roomId, (players) => {
  const host = players[room.meta.hostId];
  if (host && Date.now() - host.lastSeen > 60_000) {
    tryBecomeHost(roomId, players);
  }
});

// 4. Transaction으로 새 호스트 결정 (가장 먼저 시도한 플레이어가 호스트)
async function tryBecomeHost(roomId: string, players: Record<string, Player>) {
  const metaRef = db.ref(`games/${roomId}/meta`);
  await metaRef.transaction((meta) => {
    if (!meta) return meta;
    const oldHostLastSeen = players[meta.hostId]?.lastSeen ?? 0;
    if (Date.now() - oldHostLastSeen > 60_000) {
      // 플레이어 목록에서 나 다음 순서를 새 호스트로
      const newHostId = getNextHost(players, meta.hostId);
      return { ...meta, hostId: newHostId };
    }
    return; // 취소 (다른 누군가가 이미 처리)
  });
}
```

## 9. 투표 알고리즘 상세

투표의 핵심: "좀비가 있는 구역에서 가장 적게 지지받은 플레이어의 생존자들이 희생된다"

```typescript
function calculateVoteResult(
  zone: ZoneName,
  votes: Record<string, string>,    // voterId → targetPlayerId
  state: GameState,
  tieBreakRule: TieBreakRule
): VoteResult {
  // 1. 이 구역에 생존자가 있는 플레이어만 투표에 포함
  const playersInZone = getPlayersInZone(zone, state);

  // 2. 투표 집계 (타겟 받은 횟수)
  // 투표권 = 그 구역의 내 생존자 수
  const tally: Record<string, number> = {};
  for (const playerId of playersInZone) tally[playerId] = 0;

  for (const [voterId, targetId] of Object.entries(votes)) {
    const voterSurvivors = getSurvivorsInZone(voterId, zone, state);
    const voteWeight = voterSurvivors.length; // 생존자 수 = 투표 가중치
    if (tally[targetId] !== undefined) {
      tally[targetId] += voteWeight;
    }
  }

  // 3. 최소 득표자 = 희생자
  const minVotes = Math.min(...Object.values(tally));
  const losers = Object.entries(tally)
    .filter(([, v]) => v === minVotes)
    .map(([id]) => id);

  if (losers.length === 1) {
    // 명확한 소수파
    return {
      zone,
      tally,
      loser: losers[0],
      sacrificedSurvivorIds: getSurvivorsInZone(losers[0], zone, state).map(s => s.survivorId),
      tieBreak: 'none',
    };
  }

  // 4. 동수 처리
  switch (tieBreakRule) {
    case 'all_eliminated':
      return { zone, tally, loser: null, sacrificedSurvivorIds: losers.flatMap(id => getSurvivorsInZone(id, zone, state).map(s => s.survivorId)), tieBreak: 'all_eliminated' };
    case 'random':
      const randomLoser = losers[Math.floor(Math.random() * losers.length)];
      return { zone, tally, loser: randomLoser, sacrificedSurvivorIds: getSurvivorsInZone(randomLoser, zone, state).map(s => s.survivorId), tieBreak: 'random' };
    case 'revote':
      // 재투표 필요 → 특수 상태 반환
      return { zone, tally, loser: null, sacrificedSurvivorIds: [], tieBreak: 'revote' };
  }
}
```

## 10. 요약: 핵심 설계 결정 사항

| 결정 | 선택 | 이유 |
|------|------|------|
| 누가 페이즈를 전환하나 | 호스트 클라이언트 | 서버 없는 환경에서 단일 권위 지점 필요 |
| 이동 봉인 방법 | Firebase Security Rules (pendingMoves) | 서버 코드 없이 접근 제한 가능 |
| Race condition 방지 | Firebase Transaction | 페이즈 전환, 호스트 마이그레이션에 한정 적용 |
| 호스트 끊김 대응 | lastSeen heartbeat + transaction | 자동 마이그레이션, 중복 방지 |
| 게임 엔진 분리 | 순수 함수 모듈 | Firebase/React 없이 독립 테스트 가능 |
| 투표 가중치 | 구역 내 생존자 수 | 원작 규칙 충실 반영 |
| 타이머 관리 | phaseDeadline 타임스탬프 | 클라이언트 시계 차이 최소화 |
