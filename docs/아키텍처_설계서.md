## 1. 프로젝트 개요

몰오브호러(Mall of Horror) 보드게임의 웹 기반 온라인 멀티플레이어 구현체.
별도 서버 없이 Firebase Realtime Database만으로 2~6인 실시간 플레이 지원.

### 기술 스택

| 영역 | 기술 |
|------|------|
| Frontend | React 18 + TypeScript + Vite |
| 실시간 DB | Firebase Realtime Database |
| 호스팅 | (미정) |
| 상태 관리 | React useState/useRef (Zustand 미사용) |
| 스타일 | Tailwind CSS |
| 테스트 | Vitest |

### 설계 원칙

- **호스트 권위 모델**: 서버가 없으므로 방장(hostId) 클라이언트가 모든 게임 로직을 처리
- **순수 함수 엔진**: `src/engine/` 내 모든 함수는 `(state, input) => newState` 형태. Firebase/React에 무의존
- **낙관적 업데이트 없음**: 모든 상태는 Firebase → 구독 → React 상태 순서로 흐름

## 3. 게임 도메인

### 3.1 구역 (ZoneName)

| 이름 | 표시명 | 주사위 | 수용 | 방어한도 | 폐쇄 |
|------|--------|--------|------|----------|------|
| `bathroom` | 화장실 | 1 | 3 | 3 | O |
| `clothing` | 옷가게 | 2 | 4 | 4 | O |
| `toy` | 장난감가게 | 3 | 4 | 4 | O |
| `parking` | 주차장 | 4 | ∞ | 0 (항상 습격) | X |
| `security` | 보안실 | 5 | 3 | 3 | O |
| `supermarket` | 슈퍼마켓 | 6 | 6 | 4 | O |

- **습격 조건**: `좀비 수 > 방어력` (주차장은 좀비 1개 이상이면 무조건 습격)
- **구역 폐쇄**: 좀비 8개 이상 + 사람 없음 → 이후 이동 불가
- **수용 초과 이동**: 정원이 꽉 찬 구역으로 이동 시 주차장으로 튕겨남

### 3.2 캐릭터 (CharacterId)

| ID | 이름 | 점수 | 방어 | 투표권 | 특이사항 |
|----|------|------|------|--------|----------|
| `belle` | 미녀 | 7 | 1 | 1 | 미녀 집결 구역 +1 좀비 |
| `toughguy` | 터프가이 | 5 | 2 | 1 | |
| `gunman` | 건맨 | 3 | 1 | 2 | 투표권 2개 |
| `kid` | 아이 | 1 | 1 | 1 | 3인 게임에서만 등장 |

- 플레이어당 캐릭터 수: 3~6인 → 3개 (`belle`, `toughguy`, `gunman`), 3인 → 4개 (+ `kid`)
- 캐릭터 인스턴스 ID: `{playerId}_{characterId}` (예: `p1_belle`)

### 3.3 아이템 (ItemId)

| ID | 이름 | 매수 | 효과 |
|----|------|------|------|
| `hidden_card` | 히든카드 | 3 | (미구현) |
| `cctv` | 보안카메라 | 3 | 주사위 결과 확인 |
| `sprint` | 스프린트 | 3 | (미구현) |
| `threat` | 협박 | 3 | 투표권 +1 (1회) |
| `hardware` | 하드웨어 | 3 | 방어력 +1 |
| `axe` | 도끼 | 1 | 좀비 1마리 제거 |
| `pistol` | 권총 | 1 | 좀비 1마리 제거 |
| `shotgun` | 샷건 | 1 | 좀비 2마리 제거 |
| `bat` | 야구배트 | 1 | 좀비 1마리 제거 |
| `grenade` | 수류탄 | 1 | 좀비 2마리 제거 |
| `chainsaw` | 전기톱 | 1 | 좀비 2마리 제거 |

- 게임 시작 시 덱 셔플 → 플레이어당 3장씩 배분
- 개인 아이템은 Firebase `/games/{roomId}/private/{playerId}/items/` 에 저장 (본인+호스트만 읽기)
- `playerItemCounts` (공개)로 상대방 아이템 보유 수 확인 가능

### 3.4 추가 좀비 배치 규칙

주사위 4개 배치 후, 매 라운드 보너스 좀비 2마리 추가:
1. **미녀 집결 구역** +1: 미녀가 가장 많은 구역 (동률 시 스킵)
2. **인원 집결 구역** +1: 생존 캐릭터가 가장 많은 구역 (동률 시 스킵)

## 4. 게임 페이즈 상태 머신

### 4.1 페이즈 목록 및 흐름

```
setup_place
    │ 모든 캐릭터 배치 완료
    ▼
roll_dice ◄─────────────────────────── check_win (게임 계속)
    │ 보안관이 주사위 굴림 요청
    │ (sheriffRollRequest 신호 → 호스트 처리)
    ▼
dice_reveal
    │ 3초 후 자동
    ▼
character_select
    │ 전원 캐릭터 선언 완료
    ▼
destination_seal
    │ 전원 목적지 봉인 완료
    ▼
move_execute
    │ 2초 간격으로 이동 단계별 처리
    │ 완료 후 좀비 배치
    ▼
event ──► zone_announce ──► (구역별 순환)
              │                   │
              │ 좀비 습격         │ 생존자 이벤트
              ▼                   ▼
           voting              voting
           (zombie_attack)     (truck_search / sheriff)
              │                   │
              └──────────┬────────┘
                         ▼
                      event (다음 구역)
                         │ 6개 구역 완료
                         ▼
                      check_win
                         │ 종료 조건 미충족
                         ▼
                      roll_dice (다음 라운드)
```

### 4.2 페이즈별 상세

| 페이즈 | 담당자 | 설명 |
|--------|--------|------|
| `setup_place` | 플레이어 | 주사위 2개 → 배치 가능 구역 표시 → 캐릭터 선택 배치 (보안관부터 순서대로) |
| `roll_dice` | 보안관(임시/정식) | 보안관이 버튼 클릭 → `sheriffRollRequest` 기록 → 호스트가 주사위 처리 |
| `dice_reveal` | 호스트 타이머 | 주사위 결과 공개 3초 대기. 정식 보안관만 주사위 숫자 확인 가능 |
| `character_select` | 플레이어 | 보안관부터 순서대로 이동할 캐릭터 1명 공개 선언 |
| `destination_seal` | 플레이어 | 목적지 동시 봉인 (본인+호스트만 읽기 가능) |
| `move_execute` | 호스트 타이머 | 2초마다 1명씩 이동 처리. 정원 초과 시 주차장으로 튕겨남 |
| `event` → `zone_announce` | 호스트 타이머 | 구역 1~6 순서대로 4초 대기 후 이벤트 처리 |
| `voting` | 플레이어+호스트 | zombie_attack / truck_search / sheriff 투표 |
| `check_win` | 호스트 | 승리 조건 체크 후 종료 또는 다음 라운드 |

### 4.3 보안관(Sheriff) 시스템

- `sheriffIndex`: `playerOrder` 배열의 현재 보안관 인덱스
- `isRealSheriff`: true = 정식 보안관(보안실 투표로 선출), false = 임시 보안관
- 정식 보안관: 주사위 결과(숫자)를 혼자 미리 확인 가능. 임시 보안관은 공개
- `nextSheriffPlayerId`: 보안실 투표로 결정된 다음 라운드 보안관 (null = 변경 없음)
- 라운드 종료 시 `resolveNextSheriff()`로 보안관 갱신

## 5. GameState 전체 구조

```typescript
interface GameState {
  phase: GamePhase
  round: number
  phaseDeadline: number
  phaseStartedAt: number

  // 플레이어 순서 & 보안관
  playerOrder: string[]         // 게임 내내 고정
  sheriffIndex: number
  isRealSheriff: boolean
  nextSheriffPlayerId: string | null

  // 보드
  characters: Record<string, Character>  // instanceId → Character
  zones: Record<ZoneName, ZoneState>

  // 초기 배치
  setupPlacementOrder: string[]          // 남은 배치 순서 (플레이어 ID)
  setupDiceRoll: [number, number] | null

  // 이동 페이즈
  characterDeclarations: Record<string, CharacterDeclaration>
  declarationOrder: string[]
  sealedDestinations: Record<string, SealedDestination>
  destinationStatus: Record<string, boolean>  // 봉인 완료 여부 (공개)
  resolvedMoves: ResolvedMove[]               // 이동 계획 (미리 계산)
  currentMoveStep: number                     // move_execute 현재 단계

  // 이벤트 페이즈
  lastDiceRoll: DiceRollResult | null
  currentEventZoneIndex: number               // 0~5
  currentVote: VoteState | null

  // 아이템
  itemDeck: Item[]
  itemSearchPreview: string[] | null          // 트럭 수색 프리뷰 (승자에게만)
  itemSearchWinnerId: string | null
  itemSearchChoice: { ... } | null            // 승자가 제출한 선택
  playerItemCounts: Record<string, number>    // 보유 수 (공개)

  // 신호 필드
  cardReactionWindow: CardReactionWindow | null
  sheriffRollRequest: number | null           // 보안관 주사위 요청 타임스탬프
  pendingVictimSelection: {                   // 희생 캐릭터 선택 대기
    zone: ZoneName
    loserPlayerId: string
    chosenCharacterId?: string
  } | null
  lastZombieAttackResult: {                   // 사망 공지용
    zone: ZoneName
    deadCharacterId: string
    deadPlayerId: string
  } | null

  winners: string[]
  finalScores: Record<string, number>
}
```

## 7. 호스트 처리 흐름 (hostService.ts)

호스트 클라이언트의 `GamePage.tsx`에서 `game` 상태 변화마다 `runHostStep()`가 실행되어 필요한 처리를 수행한다.

### 7.1 runHostStep 판정 트리

```
game.phase === 'roll_dice' && sheriffRollRequest
  → hostRollDice()  [주사위 굴림 → dice_reveal로 전환]

game.phase === 'character_select' && 전원 선언 완료
  → patchGameState({ phase: 'destination_seal' })

game.phase === 'destination_seal' && 전원 봉인 완료
  → hostPrepareMovement()  [이동 계획 수립 → move_execute로 전환]

game.phase === 'event' && itemSearchChoice 있음
  → hostResolveItemSearch()  → zone_announce

game.phase === 'event' && itemSearchPreview 없음
  → patchGameState({ phase: 'zone_announce' })

game.phase === 'voting' && pendingVictimSelection.chosenCharacterId 있음
  → hostResolveVote(chosenCharacterId)  → event → zone_announce

game.phase === 'voting' && 전원 투표 완료
  → zombie_attack: loserChars <= 1 → hostResolveVote(auto)
                   loserChars > 1  → pendingVictimSelection 설정
  → truck_search / sheriff: hostResolveVote()
```

### 7.2 타이머 기반 처리

```
dice_reveal  → 3초 후 → hostApplyDiceRoll() [character_select로 전환]
move_execute → 2초마다 → hostApplyNextMoveStep()
               [단계별 이동 적용 → 완료 시 좀비 배치 → event]
zone_announce → 4초 후 → 구역 이벤트 처리
               [폐쇄 / 스킵 / zombie_attack / truck_search / sheriff / 다음 구역]
```

### 7.3 핵심 hostService 함수

| 함수 | 설명 |
|------|------|
| `startGame()` | 초기 상태 생성 + 아이템 배분 + Firebase 기록 |
| `hostRollDice()` | 주사위 4개 굴림 → `lastDiceRoll` + `dice_reveal` |
| `hostApplyDiceRoll()` | `character_select`로 전환 |
| `hostPrepareMovement()` | `planMovesInOrder()` 계산 → `resolvedMoves` 기록 → `move_execute` |
| `hostApplyNextMoveStep()` | 이동 1단계 적용 → 완료 시 좀비 배치 → `event` |
| `hostResolveVote()` | 투표 결과 적용 (사망/아이템/보안관) |
| `hostResolveItemSearch()` | 트럭 수색 아이템 이전 처리 |
| `hostEndRound()` | 승리 체크 → `check_win` or `roll_dice` |

## 8. 이동 페이즈 상세

### 8.1 4단계 흐름

```
character_select
  각 플레이어: declareCharacter() → Firebase game/characterDeclarations/{uid}
  전원 완료 → 호스트: destination_seal로 전환

destination_seal
  각 플레이어: sealDestination() → Firebase game/sealedDestinations/{uid}  (비공개)
                                  + destinationStatus/{uid}: true  (공개)
  전원 완료 → 호스트: hostPrepareMovement()

move_execute (2초 간격)
  resolvedMoves[0]: "OO의 XX가 YY → ZZ 이동 중..."
    → 호스트: 이동 적용 → zones/characters 갱신
  resolvedMoves[1]: ...
  완료 → 좀비 배치 (applyZombiePlacement + applyBonusZombies) → event
```

### 8.2 이동 우선순위 & 튕김

- 선언 순서(보안관 → 다음 플레이어 순)대로 처리
- 목적지가 정원 초과이면 주차장(`parking`)으로 이동 (`bumpedToParking: true`)
- `intendedZone`: 원래 목적지 (튕겨도 유지), `targetZone`: 실제 도착지

### 8.3 관련 타입

```typescript
interface ResolvedMove {
  playerId: string
  characterId: string
  fromZone: ZoneName
  intendedZone: ZoneName    // 원래 목적지
  targetZone: ZoneName      // 실제 도착지
  order: number
  executed: boolean
  bumpedToParking: boolean
}
```

## 9. 이벤트 페이즈 상세

### 9.1 구역 처리 순서

`bathroom → clothing → toy → parking → security → supermarket` (1번~6번 순서)

각 구역에 대해 4초 공지 후:

```
isClosed?     → 스킵 (다음 구역)
좀비 0?       → survivorEvent 체크
              → sheriff 투표 (보안실이면)
              → truck_search 투표 (주차장이면)
              → 이상 없음 (다음 구역)
isUnderAttack?→ zombie_attack 투표
방어 성공?    → 이상 없음 (다음 구역)
```

### 9.2 투표 종류 (VoteType)

| 타입 | 발생 조건 | 승자 효과 |
|------|-----------|-----------|
| `zombie_attack` | 좀비 > 방어력 | 패배자의 캐릭터 1명 사망 + 구역 좀비 소멸 |
| `truck_search` | 주차장 생존자 존재 | 아이템 덱 3장 중 1장 보관, 1장 증정, 1장 반환 |
| `sheriff` | 보안실 생존자 존재 | 다음 라운드 정식 보안관 선출 |

### 9.3 투표 흐름

```
VoteState 생성 → voting 페이즈
  각 플레이어: submitVote() → game/votes/{zoneId}/{uid}
  전원 완료 → 호스트: calculateVoteResult()
    동률 → 재투표 (round+1), 2회 이상이면 전체 투표
    결과 확정 →
      zombie_attack:
        캐릭터 1명이면 자동 선택 → hostResolveVote()
        여러 명이면 pendingVictimSelection 설정
          → 패배자: submitVictimChoice() → chosenCharacterId 기록
          → 호스트: hostResolveVote(chosenCharacterId)
      truck_search / sheriff: hostResolveVote()
```

### 9.4 사망 공지

`hostResolveVote()` 완료 후 `lastZombieAttackResult` 필드 갱신.
`zone_announce` UI에서 해당 구역 공지 시 빨간 박스로 표시:
> 💀 **OOO의 OOO가 사망하였습니다.**
> 좀비들이 새로운 목표를 찾아 떠납니다.

## 12. Firebase 주의사항

- `update()` (patchGameState)에 `undefined` 값 전달 불가 → 반드시 `?? null` 사용
- Firebase는 빈 배열/객체를 `null`로 저장 → `normalizeGame()`으로 구독 시 정규화 필요
- 배열을 Firebase에 저장하면 인덱스 기반 객체 `{0: ..., 1: ...}`로 반환될 수 있음 → `Object.values()`로 변환
- Security Rules에서 child rules은 부모 write 권한을 제한할 수 없음 (추가만 가능)
